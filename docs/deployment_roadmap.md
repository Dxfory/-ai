# 工笔花鸟白描临摹 MVP 落地路线图

## 当前状态

仓库已经具备 Phase 0 后端骨架：

- FastAPI 后端可运行。
- SQLite 本地开发数据库可用。
- 作品、课程、步骤、提交、反馈 API 已有。
- 工笔/写意/山水课程模板已有，但后续主线只使用工笔花鸟。
- 版权素材 `assets` API 已有，但新版 MVP 中降级为后置能力。
- 测试已覆盖基础后端流程。
- 已具备工笔花鸟 MVP 最小闭环：上传参考图、生成线稿、创建六步练习、上传每步作业、生成叠图对照、最小 Web 页面。
- 本地边缘检测结果不符合“白描”要求，已降级为本地边缘预览；正式白描路线改为 AI 图像编辑/生成接口。

当前不急着上 Docker。第一阶段优先本地快速迭代。

## Phase A：新版方向校准

目标：把项目从“大型国画临摹平台”收束成“工笔花鸟白描临摹小样”。

交付：

- 更新 MVP PRD。
- 明确山水、人物、素材库、机构版后置。
- 明确用户上传图默认仅个人学习分析。
- 明确第一版核心体验：白描线稿、步骤页、拍照上传、叠图对照。

验收：

- 团队后续开发只围绕工笔花鸟主线展开。
- 每个接口和页面都能对应到一个内测用户动作。

## Phase B：上传与白描线稿原型

目标：让用户上传图片，并生成一张可下载的白描线稿。

状态：已完成最小可用版本。

后端交付：

- `reference_uploads` 数据表。
- `line_drafts` 数据表。
- `POST /api/v1/uploads/reference`
- `POST /api/v1/line-drafts/generate`
- `GET /api/v1/line-drafts/{draft_id}`

图像处理交付：

- AI 白描稿：调用支持 image-to-image / image edit 的模型接口，输出白底黑线工笔白描。
- 本地边缘预览：仅作备用草稿，不作为正式白描。
- 参数记录：生成方式、模型名、提示词、线条参数。

前端交付：

- 上传页。
- 原图/白描预览。
- 参数滑杆。
- 下载白描按钮。

验收：

- 上传一张清晰花鸟图后，能得到可打印/可临摹的白描稿。
- 至少能处理单花、单叶组、单鸟三类简单图。

## Phase C：分步练习会话

目标：把一张白描稿变成一组一步一页的临摹流程。

状态：已完成最小可用版本。

后端交付：

- `POST /api/v1/uploads/reference`
- `POST /api/v1/line-drafts/generate`
- `GET /api/v1/line-drafts/{draft_id}`

当前模板：

1. 上传原画。
2. 生成白描稿。
3. 下载白描稿。

前端交付：

- 原画上传。
- 原画预览。
- 白描稿预览。
- 下载白描稿。

验收：

- 用户能从 Step 1 走到 Step 6。
- 每一步都有清楚的目标、操作说明和上传入口。

## Phase D：拍照上传与叠图对照

目标：用户每一步拍照后，能看到自己的图和参考图叠在一起。

状态：已完成最小可用版本。

后端交付：

- `POST /api/v1/practice-steps/{step_id}/submission`
- `POST /api/v1/practice-steps/{step_id}/overlay`
- 叠图输出文件。
- 偏差提示的基础规则。

图像处理交付：

- 上传图裁切。
- 透明度叠图。
- 简单尺寸匹配。
- 轮廓偏差可视化。

前端交付：

- 拍照/上传按钮。
- 原图、作业图、叠图三视图。
- 透明度调节。
- 继续/重画选择。

验收：

- 用户上传作业后，能直观看到轮廓和比例偏差。
- 第一版不强求 AI 精准纠错，以可视化对照为主。

## Phase E：熟人内测

目标：用真实用户验证流程是否能跑完。

内测范围：

- 5-10 个熟人用户。
- 非专业绘画背景优先。
- 每人一张简单工笔花鸟局部。

采集内容：

- 原图。
- 白描稿。
- 每一步作业图。
- 用户是否看得懂。
- 用户在哪一步卡住。
- 最终作品。
- 口头/文字反馈。

验收：

- 至少 5 个用户完成原画上传和白描稿下载。
- 能整理出第一批线稿问题。

## Phase F：模型与产品增强

目标：根据内测数据优化，而不是提前幻想复杂模型。

优化方向：

- 白描线条更干净。
- 参数默认值更适合初学者。
- 叠图对齐更稳。
- 反馈文案更像老师提醒，而不是机器评分。

后续再考虑：

- 人工修正白描稿对照数据。
- 更专业的工笔花鸟线稿模型。
- 真实照片转工笔白描。
- AI 生成图转工笔白描。

## Docker 引入时机

当前阶段不需要 Docker。

继续使用：

- FastAPI
- SQLite
- 本地上传目录
- 本地前端开发服务器

等满足以下条件再引入 Docker：

- 前端和后端都能跑通。
- 需要给别人一键部署。
- 要引入 PostgreSQL、Redis、MinIO。
- 要部署到服务器或内测环境。

Docker Compose 的目标服务：

- `frontend`
- `backend`
- `postgres`
- `redis`
- `minio`

6. 配置 GitHub 凭据后推送当前本地提交。
